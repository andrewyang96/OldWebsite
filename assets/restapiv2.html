What is an API?

Created by Andrew Yang. © 2015. v2

According to Wikipedia:

In computer programming, an application programming interface (API) is a set of routines, protocols, and tools for building software applications. An API expresses a software component in terms of its operations, inputs, outputs, and underlying types. An API defines functionalities that are independent of their respective implementations, which allows definitions and implementations to vary without compromising each other. A good API makes it easier to develop a program by providing all the building blocks. A programmer then puts the blocks together.

APIs connect the programmer (you!) to a web service. Here some cool things you can do with the help of APIs:

  • Analyze the sentiments of streaming tweets.
  • Employ facial recognition (and other computer vision stuff) to Facebook photos.
  • Pull stock data from Yahoo Finance.
  • Automatically SMS your contacts using Twilio.
  • Gather the top search results of a Google search.
  • The possibilities are endless!

The Basics

We'll be dealing with a subset of APIs called REST APIs. REST stands for Representational State Transfer. It is used to create scalable web services. REST APIs typically communicate over HTTP (Hypertext Text Transfer Protocol). The client (a program or a user) calls a certain URL (Uniform Resource Locator) and receives some data or an error. Each URL has a domain name, and path, and sometimes parameters, as illustrated below.

mydomain.com/path/to/something?param1=hello?param2=world

The Methods, The Request, and The Response

Each call to a certain URL is executed with an HTTP method. There are many HTTP methods, but we'll be primarily dealing with four of them. These are:

  • GET - Requests a specific resource from the specified URL.
  • POST - Requests that the server accept the sent data as data at the URL specified.
  • DELETE - Requests that the server delete a specific resource.
  • PUT - Requests that the server replace a resource with another resource.

Each REST API call sends a request and receives a response. The request contains the URL the program wants to access and, if necessary, additional data and parameters. The response contains HTTP headers, which specifies details such as caching (storing data for future use) and a status code. Here is a list of some common status codes:

  • 200 - OK. Everything is alright.
  • 400 - Bad request. The server thinks the request is malformed in some way.
  • 401 - Unauthorized. You'll need to provide credentials (e.g. API key, password) to proceed.
  • 404 - Not found. Your URL is most likely bad.
  • 500 - Internal server error. You should contact whomever manages this API.

Some data is usually sent in the response. The data is usually formatted with one of these two methods:

  • XML - eXtensible Markup Language
  • JSON - JavaScript Object Notation

Idempotence

Idempotence. That sounds scary. But it's really quite simple. Idempotence is the property of certain operations that can be applied multiple times without changing the result. In math, multiplying a number by 1 is an idempotent operation. The same goes with adding 0 to a number.

Now apply this to HTTP methods. We will examine the four main HTTP methods to see whether they have this property:

  • GET: This method only fetches data from the server. It doesn't alter anything server-side and repeating a specific GET call doesn't have any unintended consequences. This method is idempotent.
  • POST: This method posts something to server, thereby changing its state in some way. If a specific POST call is repeated, then there will be unintended consequences; for example, an order may be placed multiple times or a new account will be registered multiple times. This method is NOT idempotent.
  • DELETE: This method deletes something from the server. What happens if a specific DELETE call is repeated? The first call will delete the resource as intended, but subsequent calls will not change anything as the targeted resource has already been deleted. This method is idempotent.
  • PUT: This method updates something on the server. Repeating a specific PUT call will not have unintended consequences as only the first call changes the resouce. Any subsequent calls are redundant. This method is idempotent.

As we can see, only the POST method is not idempotent. This explains why browsers give a warning when you refresh the page after submitting a form. The refresh button in the browser repeats the last call made, and forms often call an API using the POST method. Ignoring this warning can lead to unintended consequences.

Restrictions on API Usage

Some APIs restrict the number and/or frequency of calls you can make. For example, Reddit requires programs wait at least 2 seconds between each API call, and Google throttles API access once your program exceeds a certain number of calls in a 24-hour period.

The most common ways REST APIs use to keep track of usage is through API keys. If you're using an API that requires an API key for each call, you must first authenticate your program. This usually means that you must include it as a parameter in the URL, however this. And if you happen to reach the limit, you'll receive a status code indicating so. This status code varies from API to API, so check the documentation.

Putting It All Together

To put all the concepts together, let's look at the API documentation for a hypothetical service called Bloggr. Bloggr allows anyone to post notes to its online service without an API key, but it requires an API key for some calls.

GET /messages
Fetches a list of all messages. Must provide API key in the parameters!

GET /messages/{id}
Fetches a message with the specified id. (Replace {id} with a number.)

POST /messages
Post a message to Bloggr. The request must contain a message body (encoded as a parameter).

DELETE /messages/{id}
Delete the message from Bloggr with the specified id.

PUT /messages/{id}
Replaces the message with the specified id with another message.
The request must contain a message body (encoded as a parameter).
